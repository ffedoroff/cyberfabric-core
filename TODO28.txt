### TODO-04: Wrap hierarchy mutations in SERIALIZABLE transactions
- **Priority**: HIGH
- **Type**: code
- **Checklist ID**: PERF-CODE-002 / Design alignment
- **Status**: [ ] pending
- **References**:
  - `resource-group/src/domain/group_service.rs:84` -- `let conn = self.db.conn()?;` in `create_group` with no transaction wrapping
  - `resource-group/src/domain/group_service.rs:247` -- `let conn = self.db.conn()?;` in `update_group` with no transaction
  - `resource-group/src/domain/group_service.rs:343` -- `let conn = self.db.conn()?;` in `move_group` with no transaction
  - `resource-group/src/domain/group_service.rs:389` -- `let conn = self.db.conn()?;` in `delete_group` with no transaction
  - `docs/DESIGN.md:711-713` -- "authoritative invariant checks MUST run inside the same write transaction that applies closure/entity mutations" and "write transactions MUST use SERIALIZABLE isolation"
- **Issue**: No explicit transaction boundaries for hierarchy mutations. DESIGN requires SERIALIZABLE isolation for create/move/delete group operations to prevent phantom reads and ensure closure table consistency. Each mutating method calls `self.db.conn()?` and executes multiple sequential queries (resolve type, validate parent, insert group, insert closure rows) without transaction wrapping. Concurrent create/move operations can create inconsistent closure tables or cycles.
- **Proposal**: Wrap each hierarchy-mutating operation (create_group, update_group with parent change, delete_group, move_group) in a SERIALIZABLE transaction. Use `conn.begin()` or the framework transaction API. Add bounded retry (max 3) for serialization conflicts per DESIGN specification.
